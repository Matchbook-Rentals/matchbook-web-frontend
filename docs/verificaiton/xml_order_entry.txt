This document is for developers integrating alternate order entry methods into the Accio Data

can use it to develop an alternative web-based ordering platform, to integrate into an applicant

Chapter 2
Communication
2.1

SNI

Communication with Accio Servers requires SNI (Server Name Identification) aware clients. Most
modern clients support SNI, but some older software may not.

2.2

Ciphers

Communication with Accio Servers require modern communication ciphers. This may limit the
ability of older clients and servers in older environments to connect to the Accio Servers. Some
environments like Java 6, Java 7, windows XP, and the like, may not support modern ciphers.

2.2.1

Protocol

Communication with Accio Data is via XML transferred over HTTPS. Documents are posted to
the Accio Data server with a Content-Type of text/xml and are returned to you with the same
Content-Type.
Files are posted to the URL for the server of the company you’re working with. If you’re integrating with more than one company using Accio Data ’s platform, you will interact with each on
that company’s server URL. XML files are posted to the server at the URL
https://somecompany.acciodata.com/c/p/researcherxml

The URL domain name may be .acciodata.com, .bgsecured.com, or a custom URL.

The XML file sent to Accio Data has the following major components :
• login : Your account information

Chapter 3
Placing an order
To place an order, the XML file you send to Accio Data will contain one root node named New Order
containing two subnode types :
• login : Your account information
• mode : PROD—DRAFT—TEST
• placeOrder : an order (more than one may be attached)

3.1

The Login Block

<login>
<account>accountname</account>
<username>username</username>
<password>yourpassword</password>
</login>
This block is required for all requests. The account, username, and password will be assigned
to you by the company you’re working with following their administration guide’s instructions in
the “Setting up Vendor (Researcher) Accounts” section.

3.2

The Mode Node

<mode>PROD</mode>
Mode defaults to TEST, valid values are TEST, DRAFT, or PROD. Transactions submitted in
TEST mode go through the same validation as production mode posts, and return the same XML
response document as PROD posts — order numbers and all. However, in TEST mode the orders
are not committed.
DRAFT mode orders will return XML to you the same as TEST or PROD mode, but like TEST
mode orders will not be dispatched to suppliers. Unlike test mode, the order will show up in the
system in draft mode. TEST mode orders don’t show up except via Screenings / Look Up Orders —
if you have the order number — and can be committed and become production orders, or deleted
through the Accio UI.

When writing an integration, orders should be placed in TEST mode while testing your XML
interface. Once you’re done with initial testing, move to DRAFT mode. Draft mode orders will
persist on the remote system, allowing you to log in via the web interface and review the order to
ensure that the XML you sent created the intended order.
Before posting PROD mode orders, you should coordinate with your screening services provider
to ensure that orders are appropriately discarded or processed as production orders.

When sending in orders, we recommend sending one order per XML post. This makes the
debugging process easier if errors occur, and it also makes it easier to identify and separate out any
bad orders.
If you do send more than one order per post, orders will be processed in document order until
all have been processed or the first error is found. Your confirmation XML document will confirm
all placed orders and, if an error occurred, will indicate the type of error in an <error> XML node.
Because the parser stops processing at the first error, you will never receive more than one error
message in a response document.
If using the number attribute on the placeOrder node, the number supplied must be an integer.
This number will not be quoted back in the placeOrder response, but it will be quoted back in any
results that are posted to your server in the <completeOrder> node as follows :
<completeOrder number=’’123’’ remote_number=’’555444’’ ...>
number is the number that you supplied when placing the order, and remote_number is the
order number assigned in the remote system and quoted to you in the placeOrder response.
The profile is not used internally by Accio, but can be used by those vendors receiving Accio
xml however they see fit. When generating outbound Accio XML this value can be set on site setup
under Vendor Configuration Ü Accio Integrated using the <Vendor Name> Profile field. You can
also reach this setting from the vendor configuration page using the System Level variables gear
icon at the bottom of the page.
NOTE This is intended to be used by third party vendors as they see fit. Set this value as
directed by the third party vendor.

Chapter 5
Sending Imaged Documents
Sometimes it is necessary to attach images to background check requests or results. Signed release
forms, employment applications, and supporting documentation may be attached to the order or
results as images. Each order/results (<placeOrder> or <postResults>) block in the XML document can contain any number of base64 encoded images in separate <image> tags identified
by two keys (documenttype and pagerange) provided as attributes of the image tag. Each image
uploaded must have a unique combination of these keys.
Documenttype may be an arbitrary string, but there is one special document type of “Release
Form” which should be used to designate a generic release form signed by the applicant.
The tag attribute to designate whether this document is visible to clients or not is titled “clientvisible”. Valid values are ‘Y’ and ‘N’. If clientvisible is not used, the default is ‘Y’.
A sample image tag is as follows :
<image documenttype=‘Release Form’ pagerange=‘all’ format=‘tiff’ clientvisible=‘Y’>
... base64 encoded data ...
</image>
Or, to upload images in file per page mode :
<image documenttype=‘Employment Application’ pagerange=‘1’ format=‘tiff‘>
... base64 encoded data ...
</image>
<image documenttype=‘Employment Application’ pagerange=‘2’ format=‘tiff’>
... base64 encoded data ...
</image>
<image documenttype=‘Employment Application’ pagerange=‘3’ format=‘tiff’>
... base64 encoded data ...
</image>

5.1

Image File Formats

The uploaded image’s file format is designated in the “format” attribute of the image tag. The only
supported formats at this time are tiff and pdf.

Packages

If desired, XML can be used to pull a list of packages available to an account. Attempting to place
an order with an invalid packagename will result in an error message listing the available packages,
but this query will return more information about the packages available to you. All you need to
do is include a getPackages node in your XML request as in the following example :
<New_Order>
<login>
<account>youraccount </account>
<username>youruser </username>
<password>yourpassword </password>
</login>
<getPackages account="subaccount"/>
</New_Order>
The response document will be as follows :
<XML>
<package>somename</package>
.
.
.
</XML>
If the account is a sysop or client user with switch user capabilities, then you can include an
account attribute to the getPackages node.

Results for a specific order

Normally order results are pushed when a component completes and/or when the order is completed. If desired you can pull the results for a specific order. All you need to do to get the results
for a specific order is put a getOrderResults node in your XML request as in the following example

Chapter 7
Standard Data Formats
7.1

Dates

Most dates sent to Accio Data must be formatted in our standard date format, which is YYYYMMDD.
Any unknown or unspecified date element will be represented by 00. E.G. 19820300 reads as March
??, 1982.
There are also some special date formats allowed as follows :
• current : used in End Date field for present employer
• 99990000 : used in End Date field for present employer, means “current”
This is supported for both standard data entry and applicantcheck/schoolcheck. A current date
in numeric representation in the XML interface would be entered as “99990000”.
